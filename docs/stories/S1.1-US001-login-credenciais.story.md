# US-001 — Login com Credenciais Institucionais

**ID:** US-001
**Sprint:** S1 (Semanas 1–2)
**Status:** Ready
**Prioridade:** Alta
**Depende de:** — (nenhuma)
**Estimativa:** 8 pontos
**Módulo:** Autenticação e Controle de Acesso

---

## Descrição

> Como **servidor municipal**, quero fazer login com minhas credenciais institucionais (e-mail e senha), para que eu possa acessar o sistema de gestão de ordens de serviço de forma segura e rastreável.

## Contexto e Motivação

Esta é a US fundacional do sistema. Sem autenticação real, nenhuma outra funcionalidade pode existir com segurança. Substitui o mock de login existente no front-end por autenticação JWT real contra o banco PostgreSQL via FastAPI.

---

## Critérios de Aceitação

### Cenário 1: Login com credenciais válidas
```gherkin
Given que sou um servidor municipal cadastrado no sistema
When informo meu e-mail e senha corretos e clico em "Entrar"
Then sou autenticado com sucesso
And recebo um access_token (8h) e refresh_token (24h)
And sou redirecionado para o dashboard correspondente ao meu perfil
And a tentativa de login é registrada em audit_logs com ação "LOGIN"
```

### Cenário 2: Credenciais inválidas
```gherkin
Given que sou um usuário tentando fazer login
When informo e-mail ou senha incorretos
Then recebo a mensagem "Credenciais inválidas. Verifique seu e-mail e senha."
And a tentativa é registrada em audit_logs com ação "LOGIN_FAILED"
And o contador de tentativas do usuário é incrementado
```

### Cenário 3: Bloqueio por excesso de tentativas
```gherkin
Given que realizei 4 tentativas de login inválidas
When realizo a 5ª tentativa inválida
Then minha conta é bloqueada por 15 minutos
And recebo a mensagem "Conta bloqueada temporariamente. Tente novamente em 15 minutos."
And tentativas durante o bloqueio retornam a mesma mensagem sem decrementar o timer
```

### Cenário 4: Conta bloqueada e período expirado
```gherkin
Given que minha conta está bloqueada
When o período de 15 minutos expira
And realizo uma tentativa com credenciais corretas
Then sou autenticado com sucesso
And o contador de tentativas é zerado
```

### Cenário 5: Primeiro acesso — troca de senha obrigatória
```gherkin
Given que é meu primeiro acesso ao sistema (first_login = true)
When realizo o login com credenciais válidas
Then sou redirecionado para a tela de redefinição de senha obrigatória
And não consigo acessar nenhuma outra rota antes de redefinir a senha
```

### Cenário 6: Token expirado — renovação automática
```gherkin
Given que estou autenticado com um access_token expirado
When realizo qualquer requisição à API
Then o interceptor Axios tenta renovar automaticamente via refresh_token
And se o refresh for válido, a requisição original é reenviada com sucesso
And se o refresh também expirou, sou redirecionado para /login
```

### Cenário 7: Logout
```gherkin
Given que estou autenticado no sistema
When clico em "Sair"
Then a sessão é invalidada no back-end
And o token é removido do authStore
And sou redirecionado para /login
And o evento é registrado em audit_logs com ação "LOGOUT"
```

## Regras de Negócio

| ID | Regra |
|----|-------|
| RN-1 | Máximo de 5 tentativas de login antes do bloqueio temporário por 15 minutos |
| RN-2 | Access token com expiração de 8 horas (jornada de trabalho) |
| RN-3 | Refresh token com validade de 24 horas |
| RN-4 | Senha deve ter no mínimo 8 caracteres, com letras e números |
| RN-5 | Primeiro acesso exige redefinição obrigatória de senha (first_login = TRUE) |
| RN-6 | Todas as tentativas de login (sucesso e falha) devem ser registradas em audit_logs |
| RN-7 | Senhas armazenadas com bcrypt |
| RN-8 | Token JWT contém: sub (user_id), role, secretaria_id, exp |

## Escopo

### IN ✅
- Tela de Login com campo de e-mail e senha
- Autenticação via `POST /api/auth/login`
- Geração de JWT (access + refresh)
- Bloqueio por tentativas (5 tentativas / 15 min)
- Primeiro acesso com redefinição de senha
- Interceptor Axios para renovação automática de token
- Logout com invalidação de sessão
- Registro em audit_logs de todas as tentativas

### OUT ❌
- Login social (Google, Microsoft, etc.)
- Autenticação de dois fatores (2FA)
- Recuperação de senha via e-mail (escopo de US-014)
- SSO corporativo
