# US-002 — Controle de Acesso por Perfil (RBAC)

**ID:** US-002
**Sprint:** S1 (Semanas 1–2)
**Status:** Ready
**Prioridade:** Alta
**Depende de:** US-001
**Estimativa:** 8 pontos
**Módulo:** Autenticação e Controle de Acesso

---

## Descrição

> Como **administrador do sistema**, quero que cada usuário acesse apenas as funcionalidades correspondentes ao seu perfil, para que o sistema respeite as atribuições e responsabilidades de cada setor da prefeitura.

## Contexto e Motivação

O RBAC é a espinha dorsal do sistema. Sem controle de acesso por perfil, qualquer usuário poderia acessar qualquer funcionalidade — invalidando todo o fluxo de aprovação. Esta US garante que o back-end valide o perfil em cada requisição e o front-end renderize apenas as opções permitidas.

---

## Critérios de Aceitação

### Cenário 1: Usuário secretaria acessa apenas suas telas
```gherkin
Given que estou autenticado com perfil "secretaria"
When navego pelo sistema
Then vejo apenas: Dashboard simplificado, Nova Ordem, Minhas Ordens, Devolvidas, Histórico próprio
And não vejo links para: Gabinete, Controladoria, Empenho, Pagamento, Admin
And tentativa de acesso direto via URL retorna redirecionamento para /acesso-negado
```

### Cenário 2: Endpoint protegido por perfil retorna 403
```gherkin
Given que estou autenticado com perfil "secretaria"
When envio uma requisição para POST /api/ordens/:id/acao com ação "autorizar"
Then recebo HTTP 403 com detail "Acesso negado para este perfil"
And nenhuma ação é executada no banco de dados
```

### Cenário 3: Admin tem acesso somente-leitura às áreas operacionais
```gherkin
Given que estou autenticado com perfil "admin"
When acesso as telas de Gabinete, Controladoria, Empenho, etc.
Then posso visualizar todas as informações
And os botões de ação (autorizar, empenhar, etc.) estão ocultos para mim
```

### Cenário 4: Token JWT contém o role correto
```gherkin
Given que me autentico com sucesso
When o sistema decodifica meu JWT
Then o payload contém: sub, role, secretaria_id, exp
And o front-end usa o campo role para renderização condicional de menus e botões
```

### Cenário 5: Alteração de perfil pelo admin é registrada
```gherkin
Given que sou administrador
When altero o perfil de um usuário de "secretaria" para "gabinete"
Then a alteração é salva no banco
And um registro é inserido em role_change_log com: user_id, old_role, new_role, changed_by
And o usuário afetado deve fazer login novamente para o novo token refletir o novo role
```

### Cenário 6: Botões de ação ficam ocultos (não desabilitados)
```gherkin
Given que estou autenticado com perfil "tesouraria"
When acesso a tela de Pipeline de Pagamento
Then vejo o botão "Pagar" (minha ação)
And não vejo os botões de ações de outros perfis (autorizar, empenhar, etc.)
```

---

## Regras de Negócio

| ID | Regra |
|----|-------|
| RN-7 | Perfis disponíveis: secretaria, gabinete, controladoria, contabilidade, tesouraria, admin |
| RN-8 | Um usuário pode ter somente um perfil ativo por vez |
| RN-9 | Admin visualiza todas as telas com somente-leitura às áreas operacionais |
| RN-10 | Alterações de perfil devem ser registradas em role_change_log |
| RN-11 | Token JWT deve conter o campo `role` para validação no front-end |
| RN-12 | Back-end valida o perfil em CADA requisição — nunca confiar apenas no front-end |

---

## Escopo

### IN ✅
- Middleware `require_role` no back-end validando perfil por endpoint
- `RoleGuard` no front-end ocultando rotas não permitidas
- Menu lateral com renderização condicional por perfil
- Tela `/acesso-negado` para tentativas não autorizadas
- CRUD de usuários restrito a `admin`
- Alteração de perfil com registro em `role_change_log`

### OUT ❌
- Múltiplos perfis simultâneos por usuário
- Permissões granulares dentro de um perfil (ex: "gabinete que só vê")
- Interface de gestão de permissões (perfis são fixos no sistema)
