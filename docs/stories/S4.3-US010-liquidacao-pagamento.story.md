# US-010 — Liquidação e Pagamento (Contabilidade/Tesouraria)

**Sprint:** S4 (Semanas 7–8)
**Status:** Ready

---

## Descrição

> Como **contador (liquidação)** e **tesoureiro (pagamento)**, quero registrar a liquidação do empenho e o pagamento final da ordem, para que o ciclo financeiro completo seja concluído com rastreabilidade de todos os valores e datas.

## Contexto e Motivação

As duas últimas etapas financeiras do fluxo: a liquidação (reconhecimento contábil da obrigação) e o pagamento (efetivação da transferência ao fornecedor). Após o pagamento, a ordem fica em estado terminal PAGA — somente leitura para todos os perfis operacionais.

## Critérios de Aceitação

### Cenário 1: Contabilidade registra liquidação
```gherkin
Given que estou autenticado com perfil "contabilidade"
When acesso a tela "Pipeline Liquidação"
Then vejo todas as ordens com status "AGUARDANDO_LIQUIDACAO"
And seleciono uma ordem e preencho: valor_liquidado e data_liquidacao
And confirmo a liquidação
Then a ordem avança para status "AGUARDANDO_PAGAMENTO"
And os campos valor_liquidado e data_liquidacao são gravados
And registro em ordem_historico: acao="liquidar"
```

### Cenário 2: Tesouraria registra pagamento
```gherkin
Given que estou autenticado com perfil "tesouraria"
When acesso a tela "Pipeline Pagamento"
Then vejo todas as ordens com status "AGUARDANDO_PAGAMENTO"
And preencho: valor_pago, data_pagamento, forma_pagamento (transferencia/cheque/pix)
And confirmo o pagamento
Then a ordem avança para status "PAGA" (terminal)
And registro em ordem_historico: acao="pagar"
```

### Cenário 3: Ordem PAGA é somente-leitura
```gherkin
Given que uma ordem está com status "PAGA"
When qualquer usuário operacional tenta executar qualquer ação sobre ela
Then recebo HTTP 422: "Ordens pagas são somente-leitura"
And a ordem permanece no status PAGA
```

### Cenário 4: Valor pago diferente do liquidado requer justificativa
```gherkin
Given que o valor_liquidado é R$ 9.500,00
When registro um pagamento com valor_pago de R$ 9.000,00 (diferença de R$ 500,00)
Then o sistema exibe um alerta: "Atenção: valor pago difere do liquidado. Informe a justificativa."
And o campo de justificativa se torna obrigatório
And somente após preencher a justificativa o pagamento pode ser confirmado
```

### Cenário 5: Data de pagamento não pode ser futura
```gherkin
Given que estou registrando um pagamento
When informo uma data_pagamento maior que a data atual
Then o campo exibe: "A data do pagamento não pode ser uma data futura"
And o pagamento não é confirmado
```

### Cenário 6: Campos obrigatórios de liquidação e pagamento
```gherkin
Given que estou no modal de liquidação
When submeto sem preencher valor_liquidado ou data_liquidacao
Then o formulário exibe erros nos campos obrigatórios
And nenhuma requisição é enviada ao back-end
```

### Cenário 7: Admin pode reverter ordem PAGA em caso de erro
```gherkin
Given que uma ordem está com status "PAGA" e houve erro de registro
When o Administrador executa a reversão via painel admin
Then a ordem volta ao status anterior com registro em ordem_historico
And a ação é registrada com descrição do motivo
```

## Escopo

### IN ✅
- Tela de fila de liquidação (Contabilidade — AGUARDANDO_LIQUIDACAO)
- Tela de fila de pagamento (Tesouraria — AGUARDANDO_PAGAMENTO)
- Modal de liquidação: valor_liquidado + data_liquidacao + observacao (opcional)
- Modal de pagamento: valor_pago + data_pagamento + forma_pagamento + justificativa (se diferença)
- Validação: data_pagamento não pode ser futura

- Tela de ordens pagas com filtros

### OUT ❌
- Reversão de ordem PAGA (somente Admin — fora do escopo desta US)
- Parcelamento de pagamento
- Integração com sistema bancário externo
