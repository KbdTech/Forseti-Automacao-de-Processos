# US-025 — Admin: Resetar Senha de Usuário

**ID:** US-025
**Sprint:** S10
**Status:** Draft
**Prioridade:** Média
**Estimativa:** 5 pts
**Depende de:** US-013 (Gestão de Secretarias), US-002 (RBAC)
**Módulo:** Backend / users route + Frontend / UserManagementPage

---

## Descrição

> Como **administrador**, quero poder resetar a senha de qualquer usuário pelo painel de Gestão de Usuários, para que servidores que esqueceram a senha ou estão bloqueados possam acessar o sistema novamente sem depender do próprio login.

## Contexto e Motivação

Atualmente o modal de "Editar Usuário" permite alterar nome, e-mail e status. Não existe mecanismo para reset de senha pelo admin. O único reset disponível é o fluxo de primeiro acesso (first_login=True na criação). O mecanismo adotado é: admin clica em "Resetar Senha" → backend seta `first_login=True`, `login_attempts=0`, `locked_until=None` → na próxima vez que o servidor tentar logar, é redirecionado para criar nova senha (PrimeiroAcessoPage).

## Critérios de Aceitação

### Cenário 1: Botão de reset disponível no modal de edição
```gherkin
Given que sou admin e abro o modal "Editar Usuário"
Then existe um botão "Resetar Senha" abaixo dos campos de edição
And o botão tem visual de ação destrutiva (variante destructive ou outline com destaque)
```

### Cenário 2: Confirmação antes de resetar
```gherkin
Given que clico em "Resetar Senha"
Then abre um modal de confirmação com o texto:
  "Isso obrigará [nome do usuário] a criar uma nova senha no próximo acesso."
And existe botão "Cancelar" e "Confirmar Reset"
```

### Cenário 3: Reset executado com sucesso
```gherkin
Given que confirmo o reset
When a requisição é processada
Then o backend seta: first_login=True, login_attempts=0, locked_until=None
And o admin vê toast: "Senha resetada. O usuário precisará criar nova senha no próximo login."
And o modal de confirmação fecha
And o modal de edição fecha
```

### Cenário 4: Usuário é forçado a trocar senha no próximo login
```gherkin
Given que o admin resetou a senha do usuário
When o usuário tenta logar com qualquer senha
Then o login é bem-sucedido (a senha atual ainda é válida)
And o sistema redireciona para /primeiro-acesso (PrimeiroAcessoPage)
And o usuário cria nova senha e acessa normalmente
```

### Cenário 5: Ação registrada em audit_log
```gherkin
Given que o admin completou o reset
Then existe registro em audit_logs com action='RESET_PASSWORD', user_id do admin
```

### Cenário 6: Não-admin não pode resetar
```gherkin
Given que envio PUT /api/users/{id}/reset-password sem role admin
Then o backend retorna HTTP 403
```

## Especificação Técnica

### Back-End

**Arquivo:** `backend/app/api/routes/users.py`

Novo endpoint:
```python
@router.put("/{user_id}/reset-password", status_code=200)
async def reset_user_password(
    user_id: UUID,
    db: AsyncSession = Depends(get_db),
    current_user: User = Depends(require_role('admin')),
    request: Request = None,
    background_tasks: BackgroundTasks = BackgroundTasks(),
):
    user = await db.get(User, user_id)
    if not user:
        raise HTTPException(status_code=404, detail="Usuário não encontrado")

    user.first_login = True
    user.login_attempts = 0
    user.locked_until = None

    # Registrar em audit_log
    audit_log = AuditLog(
        user_id=current_user.id,
        action='RESET_PASSWORD',
        ip_address=request.client.host if request else None,
        user_agent=request.headers.get('user-agent') if request else None,
    )
    db.add(audit_log)
    await db.commit()

    return {"message": "Senha resetada com sucesso"}
```

### Front-End

**Arquivo:** `frontend/src/pages/admin/UserManagementPage.tsx`

No `EditUserDialog` (componente interno), adicionar:

1. Novo estado:
```tsx
const [showResetConfirm, setShowResetConfirm] = useState(false)
```

2. Mutation de reset (separada da mutation de edição):
```tsx
const resetMutation = useMutation({
  mutationFn: (userId: string) =>
    apiClient.put(`/api/users/${userId}/reset-password`),
  onSuccess: () => {
    toast.success('Senha resetada. O usuário precisará criar nova senha no próximo login.')
    setShowResetConfirm(false)
    onClose()
  },
  onError: () => {
    toast.error('Erro ao resetar senha. Tente novamente.')
  },
})
```

3. Botão no formulário (abaixo dos campos existentes, antes dos botões Cancelar/Salvar):
```tsx
<Button
  type="button"
  variant="outline"
  className="w-full text-destructive border-destructive hover:bg-destructive/10"
  onClick={() => setShowResetConfirm(true)}
>
  Resetar Senha
</Button>
```

4. AlertDialog de confirmação:
```tsx
<AlertDialog open={showResetConfirm} onOpenChange={setShowResetConfirm}>
  <AlertDialogContent>
    <AlertDialogHeader>
      <AlertDialogTitle>Resetar senha de {user?.nome_completo}?</AlertDialogTitle>
      <AlertDialogDescription>
        Isso obrigará o usuário a criar uma nova senha no próximo acesso.
        A senha atual continuará válida até o próximo login.
      </AlertDialogDescription>
    </AlertDialogHeader>
    <AlertDialogFooter>
      <AlertDialogCancel>Cancelar</AlertDialogCancel>
      <AlertDialogAction
        className="bg-destructive hover:bg-destructive/90"
        onClick={() => resetMutation.mutate(user!.id)}
        disabled={resetMutation.isPending}
      >
        {resetMutation.isPending ? 'Resetando...' : 'Confirmar Reset'}
      </AlertDialogAction>
    </AlertDialogFooter>
  </AlertDialogContent>
</AlertDialog>
```
