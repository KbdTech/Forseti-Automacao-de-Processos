# US-014 — Notificações por E-mail em Mudança de Etapa

**ID:** US-014
**Sprint:** S6 (Semanas 11–12)
**Status:** Ready
**Prioridade:** Média
**Depende de:** US-005, US-007, US-009
**Estimativa:** 8 pontos
**Módulo:** Notificações

---

## Descrição

> Como **servidor de uma secretaria ou fiscal**, quero receber notificações por e-mail quando o status de uma ordem relevante para mim mudar, para que eu possa agir prontamente sem precisar verificar o sistema constantemente.

## Contexto e Motivação

As notificações por e-mail reduzem o tempo de resposta em cada etapa do fluxo, eliminando a necessidade de acompanhamento ativo. São assíncronas para não bloquear as transições de status da API. Uma falha no envio nunca impede a tramitação de uma ordem.

---

## Critérios de Aceitação

### Cenário 1: E-mail enviado após autorização do Gabinete
```gherkin
Given que o Gabinete autoriza uma ordem
When a transição de status é concluída com sucesso
Then um e-mail é disparado de forma assíncrona para o servidor da secretaria responsável
And o e-mail contém: número do protocolo, secretaria, nova etapa (AGUARDANDO_CONTROLADORIA), observação e link direto para a ordem
And a transição de status não espera o envio do e-mail para retornar resposta ao front-end
```

### Cenário 2: E-mail enviado após devolução pelo Gabinete
```gherkin
Given que o Gabinete solicita alterações em uma ordem
When a transição para DEVOLVIDA_PARA_ALTERACAO é concluída
Then um e-mail é enviado para o servidor da secretaria com o motivo da devolução destacado
And o link no e-mail leva diretamente à tela de edição da ordem devolvida
```

### Cenário 3: E-mail enviado após atesto ou recusa de nota
```gherkin
Given que a secretaria atesta ou recusa uma nota fiscal
When a transição é concluída
Then um e-mail é enviado para a Contabilidade (liquidação) ou registrado no log (recusa)
And o e-mail contém dados da ordem e próximo passo esperado
```

### Cenário 4: Falha no envio de e-mail não bloqueia transição
```gherkin
Given que o servidor de SMTP está indisponível
When uma transição de status é executada
Then a transição é concluída com sucesso no banco de dados
And a falha do e-mail é registrada em notification_log com status "falhou"
And nenhum erro é propagado para o front-end por causa do SMTP
```

### Cenário 5: Usuário pode desativar notificações
```gherkin
Given que estou na tela de preferências de notificação
When desativo o evento "ordem_devolvida"
Then não recebo mais e-mails para esse evento
And minha preferência é salva em user_notification_prefs
And outros eventos permanecem ativos
```

### Cenário 6: Admin pode gerenciar quais eventos disparam e-mail
```gherkin
Given que estou autenticado como admin
When acesso as configurações de eventos de notificação
Then posso ativar ou desativar cada tipo de evento para o sistema inteiro
And as configurações ficam persistidas no banco
```

### Cenário 7: Log de notificações registrado sempre
```gherkin
Given que qualquer evento de e-mail é disparado
When o envio é tentado (com sucesso ou falha)
Then um registro é inserido em notification_log com: ordem_id, evento, destinatário, status (enviado/falhou), created_at
And esse log é visível para o administrador
```

## Regras de Negócio

| ID | Regra |
|----|-------|
| RN-69 | E-mails são disparados de forma assíncrona (fila — não impacta a resposta da API) |
| RN-70 | Configuração de quais eventos disparam e-mail deve ser gerenciável pelo Administrador |
| RN-71 | E-mail deve conter: número do protocolo, secretaria, etapa atual, observação e link direto para a ordem |
| RN-72 | Falha no envio de e-mail não deve bloquear a transição de status no sistema |
| RN-73 | Usuário pode configurar preferência de receber ou não notificações |

## Escopo

### IN ✅
- Serviço de notificação assíncrona via FastAPI BackgroundTasks
- Envio de e-mail via SMTP configurado em .env
- Eventos cobertos: autorizar, solicitar_alteracao, irregularidade, solicitar_documentacao, atestar, recusar_atesto
- Registro em notification_log (enviado/falhou)
- Tela de preferências de notificação por usuário
- Configuração de eventos pelo Admin
- Tabelas: notification_log, user_notification_prefs

### OUT ❌
- Notificações push (browser/mobile)
- SMS ou WhatsApp
- Fila de mensagens com retry automático (Celery/RabbitMQ) — usar BackgroundTasks FastAPI
- Templates de e-mail com HTML avançado (HTML simples ou texto é suficiente)
- Recuperação de senha por e-mail
