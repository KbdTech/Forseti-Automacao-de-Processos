# US-006 — Reenvio de Ordem Devolvida pela Secretaria

**ID:** US-006
**Sprint:** S3 (Semanas 5–6)
**Status:** Ready
**Prioridade:** Alta
**Depende de:** US-005
**Estimativa:** 5 pontos
**Módulo:** Workflow de Aprovação

---

## Descrição

> Como **servidor de uma secretaria**, quero editar e reenviar uma ordem que foi devolvida pelo Gabinete para alterações, para que eu possa corrigir os pontos apontados sem perder o protocolo original e o histórico da demanda.

## Contexto e Motivação

Quando o Gabinete devolve uma ordem, a secretaria precisa ter a oportunidade de corrigi-la. O protocolo original é preservado, o histórico é mantido e a versão da ordem é incrementada a cada reenvio, garantindo rastreabilidade completa das alterações.

---

## Critérios de Aceitação

### Cenário 1: Visualizar ordens devolvidas
```gherkin
Given que estou autenticado com perfil "secretaria"
When acesso a tela "Devolvidas para Alteração"
Then vejo apenas ordens da minha secretaria com status "DEVOLVIDA_PARA_ALTERACAO"
And cada linha exibe: protocolo, data da devolução, motivo resumido, prioridade
And o motivo da devolução é destacado visualmente (borda/badge amarelo)
```

### Cenário 2: Formulário pré-preenchido com dados originais
```gherkin
Given que seleciono uma ordem devolvida
When clico em "Editar e Reenviar"
Then o formulário abre pré-preenchido com todos os dados originais da ordem
And os campos "Secretaria" e "Protocolo" estão bloqueados para edição
And o motivo da devolução é exibido em destaque acima do formulário
And todos os outros campos estão editáveis
```

### Cenário 3: Reenvio com dados corrigidos
```gherkin
Given que editei os dados necessários na ordem devolvida
When clico em "Reenviar para o Gabinete"
And confirmo no modal de confirmação
Then a ordem volta ao status "AGUARDANDO_GABINETE"
And o campo versao da ordem é incrementado (versao + 1)
And um registro em ordem_historico: acao="reenviar", status_anterior=DEVOLVIDA_PARA_ALTERACAO, status_novo=AGUARDANDO_GABINETE
And o protocolo original é mantido — nenhum novo protocolo é gerado
```

### Cenário 4: Campos secretaria e protocolo não podem ser alterados
```gherkin
Given que estou no formulário de edição de uma ordem devolvida
When tento modificar o campo "Secretaria" ou "Protocolo"
Then os campos estão desabilitados e não aceitam entrada
And o back-end ignora qualquer tentativa de alterar secretaria_id ou protocolo via PUT
```

### Cenário 5: Histórico exibe todas as versões
```gherkin
Given que uma ordem foi devolvida e reenviada 2 vezes
When acesso o histórico dessa ordem
Then vejo todos os eventos em ordem cronológica: criar → solicitar_alteracao → reenviar → solicitar_alteracao → reenviar
And o contador de versão mostra "Versão 3" na ordem atual
```

### Cenário 6: Somente ordens DEVOLVIDA_PARA_ALTERACAO podem ser editadas
```gherkin
Given que uma ordem está com status "AGUARDANDO_GABINETE"
When o back-end recebe PUT /api/ordens/:id com dados alterados
Then recebo HTTP 422: "Somente ordens com status DEVOLVIDA_PARA_ALTERACAO podem ser editadas"
```

## Regras de Negócio

| ID | Regra |
|----|-------|
| RN-32 | Somente ordens com status DEVOLVIDA_PARA_ALTERACAO podem ser editadas |
| RN-33 | Todos os campos são editáveis exceto secretaria e protocolo |
| RN-34 | O protocolo original é mantido — não é gerado novo número |
| RN-35 | Reenvio deve incrementar o contador de versão da ordem |
| RN-36 | Histórico deve mostrar todas as versões e devoluções da ordem |

## Escopo

### IN ✅
- Tela de ordens devolvidas com destaque visual do motivo (amarelo)
- Formulário de edição pré-preenchido com campos secretaria/protocolo bloqueados
- Endpoint PUT /api/ordens/:id para edição (somente DEVOLVIDA_PARA_ALTERACAO)
- Ação "reenviar" via PATCH /api/ordens/:id/acao
- Incremento automático de versao a cada reenvio
- Registro em ordem_historico com ação "reenviar"

### OUT ❌
- Edição de ordens em outros status
- Alteração de secretaria ou protocolo
- Geração de novo protocolo no reenvio
- Upload de novos anexos
