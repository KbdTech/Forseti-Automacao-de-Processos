# S9.6 — Polimentos UI/UX: Formatação, Labels e Documentos Obrigatórios

**ID:** S9.6
**Sprint:** S9
**Status:** Done
**Prioridade:** Alta
**Estimativa:** 8 pts
**Depende de:** US-003, US-004, US-006, US-013, US-020
**Módulo:** Frontend (múltiplas telas) + Backend (validação UX-003)

---

## Descrição

> Como **usuário do sistema**, quero que as informações exibidas nas tabelas e formulários estejam sempre formatadas corretamente (valores em R$, labels legíveis, nomes de secretaria sem redundância) e que os documentos de suporte sejam obrigatórios na criação de uma ordem, para ter uma experiência consistente e profissional em todas as telas.

## Contexto e Motivação

Levantamento de polimentos visuais e de usabilidade identificados na revisão da Sprint 9. Todos os itens são **front-end only**, exceto UX-003 que adiciona validação no back-end. Nenhum item exige migration de banco.

## Critérios de Aceitação

### BUG-004 — Valor Estimado formatado em R$

```gherkin
Given que acesso a tela de Ordens Devolvidas como secretaria
When a tabela é renderizada
Then a coluna "Valor Est." exibe os valores no formato "R$ X.XXX,XX"
And não exibe "3000.00" ou qualquer formato americano

Given que acesso qualquer outra tela com coluna de valor estimado
Then todos os valores monetários seguem o mesmo padrão R$ brasileiro
```

### BUG-005 — Label "Criado por" no histórico

```gherkin
Given que acesso o histórico/timeline de qualquer ordem
When visualizo uma entrada de criação da ordem
Then o texto exibe "Criado por [Nome] — [Secretaria]"
And não exibe "Criar por Servidor — [Secretaria]"
```

### BUG-006 — Nome de secretaria sem prefixo redundante

```gherkin
Given que visualizo qualquer coluna ou campo exibindo o nome da secretaria
Then o texto exibido não contém o prefixo "Secretaria Municipal de/do/da"
  Ex.: "Secretaria Municipal de Educação" → "Educação"
  Ex.: "Secretaria Municipal de Obras" → "Obras"
  Ex.: "Secretaria Municipal de Saúde" → "Saúde"
And nomes que não seguem esse padrão (ex.: "Serviço Autônomo de Água e Esgoto") são exibidos sem alteração
```

### BUG-007 — Formatação R$ em Gestão de Secretarias

```gherkin
Given que acesso a tela de Gestão de Secretarias como admin
When a tabela de secretarias é renderizada
Then a coluna "Orçamento Anual" exibe valores no formato "R$ 1.200.000,00"
And não exibe "1200000.00"

Given que abro o modal de criação/edição de secretaria
Then o campo de orçamento anual também usa máscara de moeda BRL
```

### UX-003 — Documentos de suporte obrigatórios na Nova Ordem

```gherkin
Given que estou na Etapa de documentos da Nova Ordem
When visualizo o campo "Documentos de suporte"
Then o label exibe asterisco vermelho "*" indicando campo obrigatório
And o texto "(opcional)" é removido do label e do helper text

Given que preencho todas as etapas mas não anexo nenhum documento de suporte
When chego na Etapa de Revisão e clico em "Criar Ordem"
Then o botão permanece desabilitado
And uma mensagem orienta: "Anexe ao menos um documento de suporte para criar a ordem."

Given que anexo ao menos 1 documento de suporte e preencho todos os outros campos
When clico em "Criar Ordem"
Then a ordem é criada com sucesso normalmente

Given que envio POST /api/ordens sem ter feito upload de documento de suporte via API direta
Then o backend retorna HTTP 422
And a mensagem é "Documento de suporte obrigatório para criação da ordem"
```

### BUG-006 — Nova função utilitária em `formatters.ts`

**Arquivo:** `frontend/src/utils/formatters.ts`

```ts
/**
 * Remove o prefixo "Secretaria Municipal de/do/da/dos/das" para exibições compactas.
 * Nomes que não seguem o padrão são retornados sem alteração.
 */
export function formatNomeSecretaria(nome: string): string {
  return nome.replace(/^Secretaria Municipal (de|do|da|dos|das) /i, '')
}
```

Aplicar `formatNomeSecretaria(secretaria.nome)` em todos os locais onde o nome completo é exibido em colunas de tabela, dropdowns de filtro e labels de formulário.

> **Não alterar:** o valor salvo no banco (o nome completo deve permanecer intacto). Apenas a camada de display muda.

### UX-003 — `NovaOrdemPage.tsx` + `backend/app/services/ordem_service.py`

**Front-End:**

1. Atualizar label do campo de documentos:
```tsx
// Antes
<Label>Documentos de suporte <span className="text-muted-foreground">(opcional)</span></Label>

// Depois
<Label>Documentos de suporte <span className="text-destructive">*</span></Label>
```

2. Atualizar helper text:
```tsx
// Antes
"PDF, JPEG ou PNG — máx 10 MB por arquivo. Serão anexados automaticamente ao criar a ordem."

// Depois
"PDF, JPEG ou PNG — máx 10 MB por arquivo. Obrigatório: ao menos 1 documento antes de criar a ordem."
```

3. Adicionar guarda na Etapa de Revisão / botão de submit:
```tsx
// A guarda de habilitação do botão "Criar Ordem" deve incluir:
const podeSubmeter = /* campos válidos */ && documentosSuporte.length > 0
```

4. Mensagem de orientação quando vazio:
```tsx
{documentosSuporte.length === 0 && (
  <p className="text-xs text-destructive mt-1">
    Anexe ao menos um documento de suporte para criar a ordem.
  </p>
)}
```

**Back-End:**

**Arquivo:** `backend/app/services/ordem_service.py`

Adicionar validação no método de criação de ordem, **após** o commit da ordem e **antes** de retornar o response — ou como validação pré-commit usando os documentos enviados no mesmo request (depende de como o fluxo atual funciona):

```python
async def _validar_documentos_criacao(
    self, ordem_id: UUID, db: AsyncSession
) -> None:
    """Garante ao menos 1 documento de suporte na criação da ordem."""
    result = await db.execute(
        select(Documento).where(Documento.ordem_id == ordem_id)
    )
    if not result.scalar_one_or_none():
        raise HTTPException(
            status_code=422,
            detail="Documento de suporte obrigatório para criação da ordem"
        )
```

> **Atenção:** Verificar se o upload de documentos ocorre em requisição separada (POST /ordens/{id}/documentos) após a criação da ordem. Se sim, a validação no backend não é possível no mesmo request — a guarda no front-end é suficiente. Adicionar backend apenas se o fluxo atual suportar.
